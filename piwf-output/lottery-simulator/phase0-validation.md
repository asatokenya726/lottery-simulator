## 1. コンセプト定義

**コンセプト:** 「宝くじシミュレーター」 -- 宝くじの当選確率を10連ガチャのUIで体感できる、ノーリスクのWebエンターテインメント

**ターゲット:** 20-35歳のスマホゲーム経験者で、宝くじに興味はあるが売り場に行くほどではなく、ガチャ感覚で気軽に確率を体感したい層

**課題:** 宝くじの期待値（還元率約47%）を体感で理解できる場が存在せず、既存シミュレーターは統計ツール止まりでゲーム性・継続動機がない

**キャッチコピー:** 「10連で夢を回せ。所持金ゼロでもまた明日。」

## 4. 技術フィージビリティ概算

### 想定技術スタック

| レイヤー | 技術 | 選定理由 |
|---|---|---|
| フレームワーク | Next.js (App Router) | SSG/SSRでSEO対応可能。OGP/meta対応が容易。個人開発で十分な実績 |
| 言語 | TypeScript | 型安全でバグを減らせる。個人開発の品質担保に有効 |
| スタイリング | Tailwind CSS | ユーティリティファーストで高速開発。レスポンシブ対応が容易 |
| アニメーション | Framer Motion or CSS Animations | ガチャ演出に必要。軽量で学習コスト低い |
| データ保存 | LocalStorage (MVP) | サーバー不要。予算ゼロに適合 |
| ホスティング | Vercel (無料枠) | Next.jsとの親和性が最高。無料枠で十分 |
| SNSシェア | Web Share API + OGP | X(Twitter)シェア対応。Next.jsのmetadata APIで容易 |

### MVP期間の見積もり

| フェーズ | 期間 | 内容 |
|---|---|---|
| 設計・環境構築 | 2-3日 | プロジェクトセットアップ、確率テーブル設計、画面設計 |
| コア機能実装 | 5-7日 | ゲーム内通貨管理、10連ガチャ抽選ロジック、結果表示 |
| 演出・UI | 3-5日 | ガチャ演出アニメーション、レスポンシブUI、当選演出 |
| 付随機能 | 3-4日 | 毎日補給、収支履歴、SNSシェア、注意喚起、リセット機能 |
| テスト・調整 | 2-3日 | 確率検証、ブラウザテスト、パフォーマンス調整 |
| **合計** | **約2-3週間** | 企画書の「2-4週間」に収まる |

### 技術的ブロッカーの確認

| 確認項目 | 判定 | 備考 |
|---|---|---|
| 抽選ロジック（確率再現） | 問題なし | JavaScriptのMath.random()で重み付きランダムを実装可能 |
| LocalStorage制限 | 問題なし | 5MB程度。MVPでは十分 |
| ガチャ演出アニメーション | 問題なし | CSS Animations / Framer Motionで実現可能 |
| SEO/OGP対応 | 問題なし | Next.jsのmetadata APIで標準対応 |
| SNSシェア画像生成 | やや注意 | html-to-imageやcanvasで対応可能だがデバイス互換性に注意 |
| 法的リスク（富くじ罪） | コード外の課題 | ゲーム内通貨のみ・換金不可を明示すれば問題なし |
| HTTPS通信 | 問題なし | Vercelは標準でHTTPS |
| クライアント時刻改ざん | MVP許容 | 要件定義書R-06で許容済み |

**結論: 明確な技術的ブロッカーはなし。**

### 判定

**このアイデアはやる価値がある。** 以下の理由による:

1. **市場ギャップが明確:** 「ガチャUI x 宝くじ確率 x ゲームループ」の組み合わせは未開拓
2. **技術的に実現可能:** クライアントサイド完結で予算ゼロ制約をクリア
3. **ターゲットが明確:** ソシャゲ世代に刺さるUI/UXで差別化
4. **バイラル導線あり:** 「1等当たった!」のSNSシェアが自然な拡散動機になる
5. **MVP期間が現実的:** 2-3週間で個人開発可能なスコープ
